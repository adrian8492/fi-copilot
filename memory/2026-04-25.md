## Phase 5a — Two-party recording consent (commit `6fbaa7b`)

The discovery: `LiveSession.tsx` already had a single-party consent flow — a CFPB disclosure block and a "I confirm verbal/written/electronic consent obtained" checkbox where the manager attests that they got the customer's consent. Schema field `sessions.consentObtained` plus `consentMethod` enum (verbal/written/electronic) on `dealership_settings`. The legacy gate at `server/websocket.ts:554` blocked Deepgram if `!session.consentObtained`.

For Washington two-party law (RCW 9.73.030), a manager attestation is not sufficient. The customer must also explicitly consent — and ideally with a timestamped, IP-tagged, device-fingerprinted audit trail. That's what `consent_logs` is.

Design choice: keep the legacy single-party flow as a back-compat fallback so existing tests don't break, and layer the new `consent_logs` source-of-truth on top. Both `server/websocket.ts` and `server/http-stream.ts` apply the gate in this order:

1. If a `consent_logs` row exists for the session → use it. Block if `revokedAt != null` or `recordingMode != "recording"`.
2. Otherwise → fall back to legacy `session.consentObtained`.

This means new sessions opt into the strict two-party check via the new modal flow; old sessions and existing tests keep working untouched.

### State machine

`consent_logs` is unique on `sessionId`. One row per session, evolving through these states:

- `pending` (initial) — neither party has consented
- `pending` → `recording` — both `customerConsentAt` AND `managerConsentAt` set, no revoke
- `pending` → `manager_only` — customer declined (`recordingMode = "manager_only"`, `revokedAt = createdAt`, `revocationReason = "customer_declined"`)
- `recording` → `manager_only` — mid-session revoke (`revokedAt = now`, `revocationReason = "customer_revoked_mid_session"`)

Helpers in `server/db.ts`: `getConsentLogBySession`, `recordCustomerConsent`, `recordManagerConsent`, `declineCustomerConsent`, `revokeConsent` — all upsert-pattern, with pre-existing-revoke checks so a revoked session can't be re-consented through the same row.

### tRPC router `consent`

Five procedures: `getStatus` (read-only), `recordCustomerConsent`, `recordManagerConsent`, `declineCustomer`, `revoke`. All use `assertSessionAccess(ctx, session)` to block cross-tenant — a Korum admin attempting to consent on a Paragon session gets `FORBIDDEN` before the helper is called. IP address is pulled from `ctx.req?.ip ?? ctx.req?.headers?.["x-forwarded-for"]` (req shape comes from Express through tRPC's `CreateExpressContextOptions`). Device fingerprint is the user-agent string from the client (256-char limit). Each procedure also writes an audit log via `insertAuditLog` with the resource scope.

### Frontend (`client/src/pages/LiveSession.tsx`)

Refactored `handleStartSession` (which was 165 lines doing session-create + mic + WS + recording inline) into a controlled split:

- `handleStartSession()` — creates the session via `sessions.create` (with legacy `consentObtained: true, consentMethod` for back-compat), opens the consent modal, returns. **Does not start the mic or WS.**

The consent modal renders on the setup screen via shadcn `<Dialog>`. Two big buttons (`Customer: I Consent` / `Manager: I Consent`) light up green with `<CheckCircle2>` once tapped. A small footer line cites RCW 9.73.030. The "Begin Recording" CTA is disabled until both buttons are recorded AND server returns `recordingMode === "recording"`.

For mid-session revocation: a small "Revoke Consent" pill in the live session control bar opens a second `<Dialog>` confirming the action. On confirm: calls `consent.revoke`, stops MediaRecorder, closes WebSocket, kills the MediaStream tracks, flips `consentRecordingMode = "manager_only"`, sets `isRecording = false`. The session UI continues but the recording controls disappear.

## Phase 5b — Data deletion + audit trail (commit `8c31285`)

FTC Safeguards Rule baseline: a customer (or DP on their behalf) can request data deletion. 30-day soft-delete window before a cron sweeps and hard-deletes. Cancellation allowed during the window. Plus an admin-side accessor for compliance reviews.

### Schema + helpers

`data_deletion_requests` table (migration 0023) with NOT NULL `dealershipId`, status enum (pending / completed / cancelled), `scheduledDeletionAt` (defaults to +30 days from creation), `completedAt`, `cancelledAt`, `cancelledBy`, `customerId` / `sessionId` scope columns, plus `customerEmail` / `customerName` / `reason` / `notes` for non-internal-customer cases. Indexed on `dealershipId` + `scheduledDeletionAt` (the cron sweep needs that second index).

Six helpers in `server/db.ts`: `createDataDeletionRequest`, `getDataDeletionRequestById`, `getDataDeletionRequestsByDealership`, `cancelDataDeletionRequest`, `completeDataDeletionRequest` (cron-side), `getPendingDeletionRequestsDue`. Plus `getAuditTrailForCustomer` which reads `audit_logs` scoped to a customer.

### tRPC router `dataDeletion`

`submit` (protected) — verifies cross-tenant on both `customerId` and `sessionId` before creating. Writes audit log, fires `notifyOwner` (Manus webdev notification — `.catch()`'d so failures don't break the flow; real fires only when `forgeApiKey` env is set in production).

`list` (admin-only) — returns the dealership's requests. Super-admin path explicitly returns empty (forces explicit scoping).

`getStatus` (protected) — single request lookup with cross-tenant check.

`cancel` (admin-only) — cross-tenant check + `status === "pending"` check + audit log.

### Audit trail accessor

`admin.auditTrailForCustomer` — admin-only, returns up to 500 audit_logs entries where `resourceType = "customer"` and `resourceId = String(customerId)`. Cross-tenant check on the customer record itself before query. Self-audits the read (logs that admin X read audit trail for customer Y).

## Reframes / discoveries

**Phase 5a — pre-existing single-party flow.** When I first opened `LiveSession.tsx`, I expected to be wiring consent for the first time. Reading the code flipped that: there was already a CFPB disclosure block, a verbal/written/electronic consent-method selector, and a single-checkbox attestation. The pivot was understanding that this is a manager attestation that consent was obtained out of band, not the customer's actual consent event. The new modal records the customer's tap as a separate event, with timestamp + IP + device fingerprint. Both layers coexist; the new layer takes precedence when present.

**Phase 5a — gate placement.** The first instinct was to put the gate in the tRPC `sessions.create` mutation (block creation if no consent). Wrong: consent_logs needs `sessionId` as a foreign key, so the session has to exist first. The gate has to be at the moment of audio invocation, which is the WebSocket `start_session` message and the HTTP `POST /api/session/start` handler. Both got updated.

**Phase 5c — DPA file location.** The spec said `content/legal/dpa-template-v1.md`. I initially put it in `~/asura-build/content/legal/` because that matched the path literally. But asura-build is Oliver's repo (commits authored by `Oliver <oliver@asuragroup.com>`), the DGX side doesn't have local git identity for it, and the autonomous-mode authorization was for `feature/multi-tenant-pilot` on fi-copilot. Moved the file to `~/fi-copilot/docs/legal/dpa-template-v1.md` instead — co-located with the code that enforces it, in the repo where I'm authorized to commit. Cleaner anyway.

**Phase 5b — `notifyOwner` failure mode.** First test run after wiring `notifyOwner` to the data-deletion submit path showed a stderr error: `[DataDeletion] notifyOwner failed: TRPCError: Notification service URL is not configured`. Looked alarming. Actually fine — the function throws synchronously when `forgeApiKey` env is unset, but my call is `notifyOwner(...).catch(err => console.error(...))` so the rejection is caught and the user-facing flow is unaffected. In production the env var is set; in tests the catch handler swallows it. Confirmed by the test passing.
