# Sprint Journal — 2026-04-24

## Phase 1 reframe — schema existed but enforcement was missing

**The discovery:** When I read the spec at `~/asura-build/content/briefs/fi-copilot-benstock-pilot-sprint.md`, Phase 1 was described as "build dealerships + users tables, add dealership_id to existing tables, write 20+ isolation tests." Sounded greenfield.

When I actually read `drizzle/schema.ts`, the picture flipped:

- `dealerships`, `dealership_groups`, `dealership_settings`, `user_rooftop_assignments`, `invitations` — already exist
- `users.dealershipId`, `sessions.dealershipId`, `customers.dealershipId`, `productMenu.dealershipId` — already on the schema
- `getUserAccessibleDealershipIds`, `getAllUsersByDealershipIds`, `getAllSessionsByDealershipIds`, `assignUserToRooftop`, group-admin pathways — all already built in `server/db.ts` (2264 LOC) and `server/routers.ts` (2381 LOC)

**The actual bug pattern (the missing 20%):**

The schema knew about tenancy, but enforcement was opt-in per route. Every "loose" query function I found in `db.ts` had this signature:

```ts
export async function getAllSessions(limit = 100, offset = 0, dealershipId?: number | null)
export async function getAuditLogs(limit = 100, offset = 0, userIds?: number[] | null)
export async function getEagleEyeLeaderboard(fromDate?, toDate?, dealershipId?)
```

Pass `undefined` and you get cross-tenant data. There were also session-scoped child queries with NO tenant filter at all:

```ts
export async function getTranscriptsBySession(sessionId: number)  // no dealership check
export async function getSuggestionsBySession(sessionId: number)  // no dealership check
// ...same for flags, grades, recordings, reports, checklists, objections, scorecards, recoveries
```

Plus a concrete bug spotted in `routers.ts:828` (`sessions.delete`): the access check is `role === "admin" || isSuperAdmin || session.userId === ctx.user.id` — **an admin from store A can delete a session from store B because dealership match is never checked.** Filed for Phase 2 cleanup, not patching as part of Phase 1 to keep the diff focused.

**Why this would have been catastrophic at the Brian Benstock install:**

Brian (Paragon Honda, NADA-visible) and Korum share a database. Any single forgotten `WHERE dealershipId = ?` in any of the ~40 query helpers, called from any of the dozens of tRPC routes in `routers.ts`, would mean Brian sees Korum's deals or vice versa. One leaked query in `getAllUsers()` (which returns every user across every dealership today, no filter) and the pilot ends.

The schema looking "done" was the trap. A first-pass code review or a "do all the existing tables have dealership_id?" checklist would have given a green light. Reading the actual data access layer flipped that.

**Phase 1 deliverable as it should be (Adrian's reframe):**

1. Add `dealershipId` (nullable, indexed) to the 12 child tables that lacked it — defense-in-depth so a forgotten join still has a tenant column to filter on
2. Build a query-level enforcement layer at `server/tenancy.ts`: `resolveTenantScope(user)` → `TenantScope` (with super-admin sentinel for null), `assertTenantAccess(scope, row)`, `tenantFilter(scope, col)` for Drizzle WHERE clauses
3. Write 20+ isolation tests proving Korum-scoped callers cannot read Paragon rows even when the underlying SQL is permissive

The full audit of every `routers.ts` call site (cleaning up the loose `dealershipId?` params) is Phase 1.5 / Phase 2 — too risky to land all at once against a 1,274-test baseline.

**MySQL vs Postgres divergence in spec:**

The spec brief uses Postgres syntax (`pgTable`, `uuid("id").defaultRandom()`, `jsonb`, `timestamp("at").defaultNow()`) and Phase 5 says "writes to Postgres." Reality: `drizzle.config.ts` is `dialect: "mysql"`, schema uses `mysqlTable`, IDs are `int().autoincrement()`. Adapted to MySQL — switching dialects is out of scope for a 4-day pilot. Flag for Adrian on next sync if the StoneEagle ingestion script (Phase 5) needs to write through the existing MySQL or get its own Postgres instance.
